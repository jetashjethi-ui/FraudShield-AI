SECTION 1: THE BIG IDEA — 11 DETECTION LAYERS THAT WIN THIS
================================================================================

Project Name: FraudShield AI
Tagline: "See the fraud. Understand the why. Stop it before it happens."

Core Concept:
    An intelligent fraud detection system with 11 distinct detection layers 
    that doesn't just FLAG fraud — it EXPLAINS why, ADAPTS to seasonal 
    spending, catches SIM swaps BEFORE the fraud, profiles risky merchants, 
    spots mule networks, detects first-transaction abuse on new accounts, 
    and recommends proportional authentication based on risk severity.

    No other team will have this many innovative layers. That's your moat.

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

11 INNOVATION LAYERS:
━━━━━━━━━━━━━━━━━━━━━

LAYER 1: EXPLAINABLE AI (XAI)
   Most fraud models are black boxes. Ours tells you WHY.
   For every flagged transaction, generate a human-readable explanation:
   "Flagged because: (a) amount is 4.7x user average, (b) occurred at 
   3:12 AM, (c) merchant category never used by this customer before."
   → Judges LOVE explainability. This alone sets you apart from 70% of teams.

LAYER 2: BEHAVIORAL FINGERPRINTING ("Spending DNA")
   Create a unique spending profile for each customer:
   * Time-of-day patterns (when do they usually pay?)
   * Amount distribution (what's their normal spend range?)
   * Merchant category preferences (where do they shop?)
   * Transaction velocity (how frequently do they transact?)
   Any deviation from this "DNA" raises the risk score.

LAYER 3: SIM SWAP EARLY WARNING 🇮🇳
   India-specific and very relevant to UPI fraud.
   Detect sudden device + SIM changes as a PRE-FRAUD signal.
   If device fingerprint changes right before a high-value transaction, 
   escalate risk massively. Catches account takeover BEFORE the fraud.
   Implementation:
   - Track device_info changes per card/user
   - "device_changed_recently" = 1 if device differs from last 5 txns
   - If device changed AND high amount → massive risk boost

LAYER 4: SEASONAL & CONTEXTUAL BASELINE SHIFTING
   Standard anomaly detection FAILS during Diwali, Black Friday, New Year 
   when everyone genuinely spends 5x normal. Our system maintains 
   contextual baselines per user per time-period:
   "This user historically spends 4x more in December"
   → Legitimate surge won't trigger false alarms.
   → Shows judges you understand PRODUCTION reality, not just lab accuracy.
   Implementation:
   - month_approx feature from TransactionDT
   - Per-card per-month average spending via Aggregate + Join
   - Compare current txn against same-period historical average

LAYER 5: ADAPTIVE STEP-UP AUTHENTICATION
   Instead of binary approve/block, recommend PROPORTIONAL responses:
   - Risk 0-30 (GREEN):  Auto-approve, zero friction
   - Risk 31-50 (YELLOW): Quick PIN re-verification
   - Risk 51-70 (ORANGE): Biometric authentication required
   - Risk 71-100 (RED):   Block + explain + notify customer
   → Reduces false positives (the #1 complaint from banks)
   → Shows practical UX thinking that judges value

LAYER 6: MERCHANT RISK PROFILING
   Most fraud systems focus on the SENDER. We also profile MERCHANTS:
   - A suddenly-appearing merchant receiving thousands of micro-txns = fraud
   - Build a dynamic merchant risk score feeding into transaction risk
   Implementation:
   - Aggregate: fraud_rate per ProductCD / merchant category
   - Aggregate: txn_count per merchant in recent window
   - Flag merchants with abnormal activity spikes

LAYER 7: SHARED DEVICE DETECTION (Mule Network Fingerprint)
   Flag accounts operated from the SAME physical device across multiple 
   identities — a classic money mule network fingerprint.
   Implementation:
   - Count: unique cards per DeviceInfo
   - Count: unique emails per DeviceInfo
   - If device is shared across 3+ identities → high risk

LAYER 8: DORMANT ACCOUNT HIJACK DETECTION
   An account inactive for months suddenly transacting at high volume 
   = near-certain account takeover.
   Implementation:
   - user_txn_count per card (from aggregate)
   - is_low_activity_user = if(txn_count < 5, 1, 0)
   - If low activity + high amount → extreme risk flag

LAYER 9: 🆕 ROUND AMOUNT SUSPICION SCORING
   Fraudsters almost always transact in perfectly round numbers 
   (₹10,000, ₹50,000, ₹1,00,000). Legitimate daily spending has 
   decimals and odd amounts (₹437.50, ₹2,199). 
   Score suspicion HIGHER for perfectly round amounts, especially 
   when combined with other risk signals.
   Implementation:
   - amount_decimal = TransactionAmt - floor(TransactionAmt)
   - is_round_amount = if(amount_decimal == 0, 1, 0)
   - is_suspicious_round = if(is_round_amount == 1 AND TransactionAmt >= 5000, 1, 0)
   - Round amount + night + unusual device = very high risk composite
   → Simple feature, but devastating when combined with other layers.

LAYER 10: 🆕 MERCHANT CATEGORY MISMATCH DETECTION
   A salaried user who always buys groceries and pays bills suddenly 
   paying a jewellery wholesaler or a crypto exchange = highly anomalous.
   This catches social engineering scams where victims are directed to 
   make unusual purchases.
   Implementation:
   - Aggregate: list of distinct ProductCD values per card1 (user's 
     historical merchant categories)
   - Count: unique merchant categories per user → user_merchant_diversity
   - is_new_category = if current ProductCD not in user's history → 1
   - If is_new_category AND high amount → elevated risk
   → In RapidMiner: Aggregate unique ProductCD count per card1, then 
     compare current txn's ProductCD against common categories.

LAYER 11: 🆕 NEW ACCOUNT HIGH-VALUE FIRST TRANSACTION FLAG
   A brand-new account's very first transaction being above ₹50,000 
   is one of the STRONGEST fraud signals. Legitimate users typically 
   start with small transactions to test the system.
   Implementation:
   - user_txn_count per card1 (already computed for behavioral features)
   - is_first_txn = if(user_txn_count == 1, 1, 0)
   - first_txn_high_value = if(is_first_txn == 1 AND TransactionAmt > 500, 1, 0)
     (using 500 as threshold since IEEE dataset uses USD amounts)
   - If first ever transaction AND high value → extreme suspicion
   → Trivially easy to implement; incredibly effective.
